# DirectX Graph Rendering Strategies

## 📋 Executive Summary

WPF.UI DirectX 그래프 시스템은 두 가지 상반된 요구사항을 동시에 만족해야 합니다:

1. **실시간 모니터링**: 오실로스코프처럼 최신 데이터만 빠르게 렌더링 (IBA TFR 스타일)
2. **데이터 분석**: 축적된 모든 데이터를 저장하고 줌/팬으로 탐색

이 문서는 기술적 접근 방식의 차이와 최적화 전략, 그리고 구현 로드맵을 제시합니다.

---

## 🎯 핵심 인사이트: IBA TFR의 성능 비결

### GPU vs CPU가 아닌, 렌더링 알고리즘의 차이

**잘못된 가정:**
> "IBA TFR은 OpenGL을 사용해서 빠르다"

**실제 이유:**
> **"화면 픽셀 수만큼만 렌더링하기 때문에 빠르다"**

### 성능 차이의 근본 원인

| 시스템 | 렌더링 연산량 | 데이터 의존성 |
|--------|-------------|-------------|
| **IBA TFR** | O(화면 픽셀 수) = O(800) | ❌ 없음 |
| **현재 WPF.UI** | O(데이터 포인트 수) = O(10,000+) | ✅ 있음 |

**결과:**

- IBA: 10,000개 데이터 → 800번 계산 → 12.5배 효율
- 현재: 10,000개 데이터 → 10,000번 계산 → 비효율

### PDA/임베디드에서도 빠른 이유

CPU만으로도 고속 렌더링이 가능한 이유:

```
전략 A (현재): 버텍스 기반 렌더링
─────────────────────────────────
데이터: 10,000 포인트
  ↓
GPU: 10,000 버텍스 변환 → 9,999 선분 래스터화
  ↓
화면: 800 픽셀 (중복 계산 발생)

연산량: O(10,000)
```

```
전략 B (IBA): 픽셀 기반 렌더링
─────────────────────────────────
데이터: 10,000 포인트 (메모리 저장)
  ↓
CPU: 800개 픽셀 × 대표 포인트 1개 선택
  ↓
화면: 800 픽셀 직접 쓰기

연산량: O(800) ← 데이터 양과 무관!
```

**핵심:** OpenGL/DirectX 여부가 아니라, 화면 해상도에 렌더링을 고정했기 때문

## 🔍 기술 스택 비교

### 렌더링 접근 방식

| 구분 | Fixed Buffer (IBA 스타일) | Dynamic Buffer (현재) |
|------|--------------------------|---------------------|
| **데이터 저장** | 고정 배열 (10K) | RingBuffer (100K) |
| **GPU 버퍼** | 초기화 시 1회 생성 | 필요 시 동적 재생성 |
| **업로드 방식** | WriteNoOverwrite (부분) | WriteDiscard (전체) |
| **렌더링 개수** | 항상 고정 (10K) | 가변 (0~100K) |
| **성능 특성** | O(고정값) | O(데이터 개수) |

### 기술 스택 구성

**공통 기반:**

- DirectX 11 (D3D11)
- HLSL Vertex/Pixel Shader
- Dynamic Vertex Buffer
- Map/Unmap 패턴

**차별점:**

- **IBA**: 픽셀 기반 샘플링 + 고정 버퍼
- **WPF.UI**: 버텍스 기반 렌더링 + 가변 버퍼

## 📊 핵심 문제: 범위 필터링과 다운샘플링 부재

### 현재 문제점

**시나리오:**

1. 60Hz × 10초 = 600,000 포인트 저장
2. 화면 너비 = 800 픽셀
3. GPU에 600,000개 버텍스 전송 (800배 과잉)

**결과:**

- CPU 부하: 데이터 병합 오버헤드
- GPU 부하: 불필요한 버텍스 처리
- 메모리: 과도한 GPU 버퍼 사용

### 필요한 솔루션

#### 1단계: 범위 필터링

- 화면에 보이는 X 범위만 추출
- 예상 효과: 데이터 80% 감소

#### 2단계: 다운샘플링

- 화면 픽셀 수만큼만 샘플링
- 예상 효과: GPU 부하 90% 감소

## 🎯 세 가지 렌더링 전략

### 전략 1: Streaming Mode (고정 버퍼)

**목적:** IBA TFR 스타일 실시간 모니터링 (최대 성능)

**기술 특징:**

- 고정 배열 (10,000 포인트)
- GPU 버퍼 1회 생성 (초기화 시)
- WriteNoOverwrite (부분 업데이트)
- 범위 쿼리 미지원

**적용 시나리오:**

- 최신 데이터만 모니터링
- 과거 데이터 조회 불필요
- 최대 성능 필요 (PDA/임베디드)

**성능:**

- 연산량: O(10,000) 고정
- 메모리: 최소
- CPU/GPU: 최소 부하

---

### 전략 2: History Mode (무제한 저장)

**목적:** 데이터 분석 (줌/팬 지원)

**기술 특징:**

- `List<T>` 무제한 저장
- 이진 탐색 기반 범위 필터링
- LTTB 다운샘플링 자동 적용
- 동적 버퍼 재생성

**적용 시나리오:**

- 모든 데이터 보존
- 과거 데이터 탐색
- 다양한 줌 레벨

**성능:**

- 연산량: O(log N + M) (N=전체, M=화면)
- 메모리: 최대
- CPU/GPU: 다운샘플링으로 최적화

---

### 전략 3: RingBuffer Mode (하이브리드)

**목적:** 범용 실시간 데이터 (현재 구현)

**기술 특징:**

- RingBuffer (100,000 포인트)
- 순환 버퍼로 오래된 데이터 자동 제거
- ⚠️ 범위 필터링 미구현
- ArrayPool 기반 병합

**적용 시나리오:**

- 최근 몇 초~몇 분 데이터
- 가끔 줌/팬
- 메모리 제한 있음

**현재 문제:**

- 100K 포인트 전체 렌더링
- 화면에 1K만 보여도 100K GPU 전송

## 📈 전략 비교표

| 특징 | Streaming | RingBuffer (현재) | History |
|------|-----------|------------------|---------|
| **용도** | 실시간 모니터링 | 범용 | 데이터 분석 |
| **데이터 저장** | 고정 (10K) | 제한적 (100K) | 무제한 (1M+) |
| **범위 필터링** | ❌ 미지원 | ⚠️ 구현 필요 | ✅ 지원 |
| **다운샘플링** | ❌ 불필요 | ⚠️ 선택적 | ✅ 자동 |
| **줌/팬** | ❌ 제한적 | ⚠️ 제한적 | ✅ 전체 |
| **성능** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **GPU 업로드** | WriteNoOverwrite | WriteDiscard | WriteDiscard |
| **렌더 개수** | 항상 10K | 가변 (0~100K) | 가변 (최대 10K) |

## 🚀 구현 로드맵

### Phase 1: 범위 필터링 (우선순위: 높음, 기간: 2주)

**목표:**

- RealTimeSeries에 `GetVisibleSpan(minX, maxX)` 구현
- 화면에 보이는 X 범위만 GPU 전송

**예상 효과:**

- GPU 업로드: 80% 감소
- CPU 부하: 50% 감소

**기술 요소:**

- 이진 탐색으로 X 범위 필터링
- RingBuffer의 두 Span 모두 필터링
- ArrayPool 기반 병합 유지

---

### Phase 2: 다운샘플링 (우선순위: 중간, 기간: 2주)

**목표:**

- 화면 픽셀 수보다 많은 데이터 자동 축소
- Uniform 또는 LTTB 알고리즘 적용

**예상 효과:**

- GPU 렌더링: 90% 감소
- 시각적 품질: 유지

**기술 요소:**

- Uniform 샘플링 (간단, 빠름)
- LTTB (시각적 특징 보존)
- 화면 해상도 기반 자동 선택

---

### Phase 3: StreamingSeries (우선순위: 낮음, 기간: 2주)

**목표:**

- 고정 버퍼 시리즈 타입 추가
- IBA TFR 스타일 렌더링

**구현 파일:**

- `StreamingSeries.cs`
- `StreamingSeriesRenderer.cs`

**기술 요소:**

- 고정 배열 (10K)
- WriteNoOverwrite
- 버퍼 재생성 없음

---

### Phase 4: HistorySeries (우선순위: 낮음, 기간: 3주)

**목표:**

- 무제한 저장 시리즈 타입
- 완전한 줌/팬 지원

**구현 파일:**

- `HistorySeries.cs`
- `HistorySeriesRenderer.cs`
- `LTTBDownsampler.cs`

**기술 요소:**

- `List<T>` 무제한 저장
- 이진 탐색 + LTTB
- 동적 다운샘플링

---

## 💡 최적화 전략 적용 순서

### 즉시 적용 가능 (Phase 1)

**목표:** 범위 필터링으로 80% 성능 개선

**시나리오:**

- 10채널 × 60Hz × 5000 points = 300만 points/sec
- 화면에 5000 points만 보임
- 현재: 100K 전체 렌더링 → 개선 후: 5K만 렌더링

**기대 효과:**

| 항목 | 현재 | Phase 1 후 | 개선율 |
|------|------|-----------|-------|
| CPU 부하 | 100% | 50% | 50% ↓ |
| GPU 부하 | 100% | 20% | 80% ↓ |
| 렌더 버텍스 | 100K | 5K | 95% ↓ |

---

### 중기 적용 (Phase 2)

**목표:** 다운샘플링으로 픽셀 기반 렌더링

**시나리오:**

- 화면 너비: 800 픽셀
- 보이는 데이터: 5000 points
- 다운샘플링: 800~1600 points로 축소

**기대 효과:**

- GPU 렌더링: 추가 70% 감소
- 시각적 품질: 손실 없음 (LTTB 사용 시)

---

### 장기 적용 (Phase 3~4)

**목표:** 모드별 최적화 완성

**Streaming Mode:**

- IBA TFR 스타일 완전 구현
- PDA/임베디드 환경 지원

**History Mode:**

- 무제한 데이터 저장
- 완전한 분석 도구

## 🔧 핵심 기술 스택

### 다운샘플링 알고리즘

| 알고리즘 | 복잡도 | 품질 | 용도 |
|---------|-------|------|------|
| **Uniform** | O(N) | ⭐⭐ | 빠른 축소 |
| **Min-Max** | O(N) | ⭐⭐⭐ | Peak 보존 |
| **LTTB** | O(N) | ⭐⭐⭐⭐ | 시각적 특징 보존 |
| **Douglas-Peucker** | O(N log N) | ⭐⭐⭐⭐⭐ | 최고 품질 |

**권장:** LTTB (성능과 품질의 균형)

### GPU 버퍼 관리

**WriteNoOverwrite (Streaming):**

- 부분 업데이트
- 버퍼 크기 고정
- 최대 성능

**WriteDiscard (RingBuffer/History):**

- 전체 교체
- 가변 크기 지원
- 유연성
